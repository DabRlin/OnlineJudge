# Online Judge 系统 - 技术栈确定方案

## 1. 最终技术栈选型

基于现代化、高性能、易维护的原则，我们确定以下技术栈：

### 1.1 后端技术栈

```yaml
语言: Python 3.11+
包管理: uv (极速包管理器)
Web框架: FastAPI 0.109+ (现代异步框架)
数据库:
  - PostgreSQL 15+ (主数据库) / SQLite (开发环境)
  - Redis 7+ (缓存 + 消息队列)
ORM: SQLAlchemy 2.0 (async)
数据迁移: Alembic 1.13
异步任务: Celery 5.3 / ARQ (FastAPI 原生)
认证: python-jose (JWT)
数据验证: Pydantic 2.5 (FastAPI 内置)
代码质量:
  - Ruff (Linter + Formatter)
  - Black (代码格式化)
  - mypy (类型检查)
测试: pytest + pytest-asyncio
ASGI服务器: Uvicorn 0.27
```

**选择理由：**
- ✅ **FastAPI 原生异步支持**，性能优异（比 Django 快 2-3 倍）
- ✅ **自动生成 API 文档**（Swagger UI + ReDoc）
- ✅ **Pydantic 数据验证**，类型安全
- ✅ **现代化设计**，代码简洁优雅
- ✅ uv 大幅提升开发效率（比 pip 快 10-100 倍）
- ✅ 适合高并发场景（判题系统）

### 1.2 前端技术栈

```yaml
语言: TypeScript 5.3
包管理: bun (极速包管理器)
构建工具: Vite 5.0 (极速构建和热更新)
运行时: 
  - 开发环境: bun (极速启动)
  - 生产环境: 静态文件（Nginx 托管）
框架: React 18.2
UI库:
  - Ant Design 5.12 (组件库)
  - TailwindCSS 3.4 (样式工具)
  - Lucide React (图标)
状态管理: Zustand 4.4
数据请求: @tanstack/react-query 5.17 + Axios 1.6
路由: React Router 6.21
代码编辑器: Monaco Editor 0.45
代码质量:
  - ESLint (代码检查)
  - Prettier (代码格式化)
测试: Vitest (Vite 原生测试框架)
```

**选择理由：**
- ✅ **Vite 极速构建**：HMR 热更新毫秒级，开发体验极佳
- ✅ **bun 包管理**：依赖安装比 npm 快 20-30 倍
- ✅ **React 生态丰富**：组件库、工具链成熟
- ✅ **TypeScript 类型安全**：减少运行时错误
- ✅ **Ant Design 企业级**：开箱即用的高质量组件
- ✅ **Vitest 原生集成**：与 Vite 配置共享，零配置测试

### 1.3 判题系统技术栈

```yaml
沙箱: Docker 24+
隔离技术:
  - Namespace (进程、网络、文件系统隔离)
  - Cgroup v2 (资源限制)
  - Seccomp (系统调用过滤)
  - AppArmor (访问控制)
消息队列: Redis (判题任务队列)
支持语言:
  - C/C++ (gcc 11, g++ 11)
  - Python (3.11)
  - Java (OpenJDK 17)
  - Go (1.21)
  - JavaScript/TypeScript (bun 1.0 / Node.js 20)
  - Rust (1.75, 可选)
```

**选择理由：**
- ✅ Docker 提供完善的隔离和资源限制
- ✅ 性能开销低，易于扩展
- ✅ 支持主流编程语言
- ✅ 安全性高

### 1.4 数据库技术栈

```yaml
关系型数据库: PostgreSQL 15
  - 主要特性: JSONB, 全文搜索, 事务支持
  - 连接池: pgBouncer
  
缓存数据库: Redis 7
  - 用途: 
    - Session 存储
    - 热点数据缓存
    - 判题队列
    - 实时排行榜 (Sorted Set)
    
文件存储: 本地存储 (开发) / MinIO (生产)
  - 用途: 测试数据、用户头像、题目附件
```

### 1.6 部署技术栈

```yaml
Web服务器: Nginx 1.24
```

## 6. 技术栈决策记录

| 决策点 | 选项 | 最终选择 | 理由 |
|--------|------|---------|------|
| 后端语言 | Python / Go / Java | **Python** | 生态成熟，开发效率高 |
| 后端框架 | Django / Flask / FastAPI | **FastAPI** | 原生异步，性能优异，自动文档 |
| 包管理器 | pip / poetry / uv | **uv** | 极速，10-100x 提升 |
| ORM | Django ORM / SQLAlchemy / Tortoise | **SQLAlchemy 2.0** | 异步支持，功能强大 |
| 前端框架 | React / Vue / Svelte | **React** | 生态最丰富 |
| JS 运行时 | Node.js / Deno / bun | **bun** | 极速，20-30x 提升 |
| UI 库 | Ant Design / MUI / Chakra | **Ant Design** | 企业级，组件丰富 |
| 数据库 | PostgreSQL / MySQL / SQLite | **PostgreSQL** (生产) / **SQLite** (开发) | 功能强大，开发便捷 |
| 沙箱方案 | Docker / LXC / VM | **Docker** | 平衡性能和安全 |

## 7. 技术栈风险评估

### 7.1 潜在风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| uv/bun 兼容性问题 | 中 | 低 | 保留 pip/npm 作为备选 |
| Docker 逃逸风险 | 高 | 极低 | 多层安全防护 |
| 性能瓶颈 | 中 | 中 | 预留优化空间 |
| 学习曲线 | 低 | 中 | 提供完整文档 |

### 7.2 技术债务控制

- 定期更新依赖版本
- 保持代码覆盖率 > 80%
- 定期进行安全审计
- 文档与代码同步更新

## 8. 总结

我们选择了一套**现代化、高性能、易维护**的技术栈：

**核心优势：**
- ⚡ **极速开发**：uv + bun 大幅提升开发效率
- 🔒 **安全可靠**：Docker 多层防护 + PostgreSQL 事务支持
- 🚀 **性能优异**：Redis 缓存 + 异步任务 + 前端优化
- 🛠️ **易于维护**：完整的类型系统 + 代码质量工具
- 📈 **可扩展**：微服务架构预留 + 水平扩展支持

这套技术栈既能满足当前 MVP 开发需求，也为未来扩展预留了充足空间！
