# Online Judge - 系统架构设计

> 最后更新：2026-03-17

## 1. 整体架构

当前采用单体架构，前后端分离部署。

```
┌──────────────────────────────────────────────────┐
│              用户浏览器                            │
└──────────────────────────────────────────────────┘
                     │ HTTP REST API
                     ▼
┌──────────────────────────────────────────────────┐
│         前端 (React SPA)                          │
│         Vite dev server :5173 / 构建后静态文件     │
└──────────────────────────────────────────────────┘
                     │ /api/v1/*
                     ▼
┌──────────────────────────────────────────────────┐
│         后端 (FastAPI)  :8000                     │
│  ┌────────────┬──────────────┬─────────────────┐ │
│  │  auth API  │ problems API │ submissions API  │ │
│  │  contests  │  users API   │  import API      │ │
│  └────────────┴──────────────┴─────────────────┘ │
│         SQLAlchemy async  +  Alembic              │
└──────────────────────────────────────────────────┘
          │ SQLite (dev)          │ Celery task
          ▼                       ▼
┌─────────────────┐    ┌──────────────────────────┐
│  oj.db (SQLite) │    │  Redis :6379             │
│  (生产: PgSQL)  │    │  - Celery broker/backend │
└─────────────────┘    └──────────────────────────┘
                                  │
                                  ▼
┌──────────────────────────────────────────────────┐
│         Celery Worker (src/judger/)              │
│         DockerJudger                             │
└──────────────────────────────────────────────────┘
                     │ docker run
                     ▼
┌──────────────────────────────────────────────────┐
│         Docker 判题容器（每次判题创建/销毁）        │
│  network=none | mem_limit | cpu_period/quota     │
│  镜像: oj-judger-python/cpp/java/go/javascript   │
└──────────────────────────────────────────────────┘
```

## 2. 模块职责

### 2.1 FastAPI 后端（`src/backend/`）

| 模块 | 路径 | 职责 |
|------|------|------|
| auth | `app/api/v1/auth.py` | 注册、登录、JWT 签发 |
| problems | `app/api/v1/problems.py` | 题目 CRUD、测试用例管理、批量导入 |
| submissions | `app/api/v1/submissions.py` | 提交、查询、rejudge |
| contests | `app/api/v1/contests.py` | 竞赛管理、报名、排行榜 |
| deps | `app/core/deps.py` | 依赖注入：db session、当前用户、权限检查 |
| security | `app/core/security.py` | JWT 编解码、密码 bcrypt |

### 2.2 Celery 判题 Worker（`src/judger/`）

| 文件 | 职责 |
|------|------|
| `celery_app.py` | Celery 应用配置，broker=Redis |
| `tasks.py` | 判题任务入口，调用 DockerJudger |
| `docker_judger.py` | Docker 容器创建/执行/结果解析 |
| `languages.py` | 各语言编译命令、运行命令、超时配置 |

### 2.3 前端（`src/frontend/src/`）

| 目录 | 职责 |
|------|------|
| `pages/` | 各页面组件（Problems、ProblemDetail、Submissions、Contests 等） |
| `services/` | API 调用封装（problem、submission、contest、auth） |
| `stores/authStore.ts` | Zustand 全局认证状态，Token 持久化 |
| `components/` | 公共组件（CodeEditor、MarkdownRenderer、Layout） |

## 3. 核心流程

### 3.1 代码提交与判题

```
用户提交代码
    │
    ▼
POST /api/v1/submissions
    │ 创建 Submission 记录（status=pending）
    │ DockerJudgeService.submit_judge_task()
    ▼
Celery 任务入队（Redis broker）
    │
    ▼
Celery Worker 消费任务
    │ 拉取题目测试用例
    │ 对每个测试用例创建 Docker 容器
    │ 比对输出
    ▼
更新 Submission 记录（status=accepted/wrong_answer/...）
    │
    ▼
前端轮询 GET /api/v1/submissions/{id}
    │ 每 1.5s 一次，最多 30 次，终态停止
    ▼
展示结果
```

### 3.2 竞赛公开接口认证

公开接口（如竞赛列表）使用 `get_current_user_optional` 依赖：
- 携带有效 Token → 返回用户对象（用于权限判断）
- 未携带 Token → 返回 `None`（不报 401）

## 4. 安全设计

### Docker 沙箱配置
```python
container = docker_client.containers.run(
    image=language_image,
    network_mode="none",      # 禁止网络
    mem_limit="256m",         # 内存上限
    cpu_period=100000,
    cpu_quota=50000,          # CPU 50%
    read_only=True,           # 根文件系统只读
    tmpfs={"/tmp": "size=64m"},  # 临时可写目录
)
```

### JWT 认证
- Access Token 有效期：7 天（开发环境）
- 算法：HS256
- 密钥：通过 `SECRET_KEY` 环境变量配置

## 5. 扩展规划

| 方向 | 当前状态 | 规划 |
|------|---------|------|
| 数据库 | SQLite | 迁移至 PostgreSQL |
| 部署 | 本地开发 | Docker Compose + Nginx |
| 判题节点 | 单节点 | 多 Worker 水平扩展 |
| 缓存 | 无 | Redis 缓存题目列表/排行榜 |
